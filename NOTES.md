# Phase 0 — Codebase Audit

## 1. How the daily phrase is generated/stored

Characters live in `characters.py` as a flat list of 600 dicts (HSK 1-3). Each entry:
```python
{"char": "你", "pinyin": "nǐ", "tone": 3, "meaning": "you",
 "example_cn": "你好吗", "example_pinyin": "nǐ hǎo ma", "example_en": "How are you?",
 "group": "Pronouns", "week": 1}
```

`send_lesson.py:get_todays_character(slot)` picks which character to send based on
`(date.today() - START_DATE)`. The selection is **deterministic** — same date + slot always
gives the same character. Nothing is persisted to a database or file after sending.

**Implication for scoring:** We do NOT need a database to look up "today's phrase." We can
call `get_todays_character(slot)` at practice time and get the same character. The natural
practice target is `example_cn` (the example sentence), since single-character pronunciation
is too short for meaningful Whisper transcription.

## 2. Where to plug in a new command

The current bot is **not persistent**. `send_lesson.py` creates a `discord.Client` in
`send_discord_dm()`, fires the DM in `on_ready`, and immediately `client.close()`. GitHub
Actions invokes it 3x/day and the process exits.

For `/practice` (slash command + message listener), we need a **separate long-running bot
process** (`bot.py`). This bot:
- Registers slash commands via `discord.py` app commands
- Listens for voice message replies
- Runs Whisper transcription (via API or locally)

The daily lesson can stay on GitHub Actions — no reason to change what works. The two systems
are independent; they share the same bot token but don't conflict (one sends DMs on a
schedule, the other listens for commands in real time).

## 3. Discord intents currently enabled

```python
intents = discord.Intents.default()
```

`default()` gives everything EXCEPT the three privileged intents:
- ❌ `presences` — not needed
- ❌ `members` — not needed
- ❌ `message_content` — **NEEDED for pronunciation feature**

The pronunciation cog needs `message_content` to read voice message attachments in replies.
This must also be toggled ON in the Discord Developer Portal → Bot → Privileged Gateway
Intents → Message Content Intent.

## 4. Database

**None.** Everything is files:
- `characters.py` — static character data (600 entries)
- `stroke_data/*.json` — downloaded from CDN by `scripts/fetch_stroke_data.py`
- `audio/*.mp3` — generated by `scripts/generate_audio.py` using edge-tts

For Phase 3+, we may want **sqlite** to track practice scores per user per date (streak
tracking, weak-spot analysis). But it's not required for the core practice loop — we can
add it in Phase 4 as polish.

## 5. Architecture decision: practice target

When a user runs `/practice`, the target phrase should be the **example sentence**
(`example_cn`) from today's character, not the single character alone. Reasons:
- Single characters are too short (~0.3s) for reliable Whisper transcription
- Example sentences exercise tones in context, which is how Mandarin is actually spoken
- The `example_pinyin` field gives us ground-truth pinyin without needing pypinyin conversion

The `/practice phrase:自定义` override lets users practice arbitrary text regardless.

## 6. Key file map

| File | Role |
|---|---|
| `send_lesson.py` | Daily DM sender (GitHub Actions entry point) — DO NOT MODIFY for practice feature |
